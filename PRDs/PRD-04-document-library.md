## 1. Overview

PRD-04 introduces the Document Library: a persistent, shared, searchable repository of all generated reports backed by a SQLite database. Every APM on the shared Hostinger VPS deployment will be able to find, preview metadata for, and re-download any report the team has ever generated.

---

## 2. Problem Statement

### Current State
Generated PDFs are saved to local folders with no centralized index. APMs cannot access reports generated by colleagues. Duplicate reports get generated for the same facility because prior work cannot be found.

### Desired State
All generated reports are automatically cataloged at moment of creation. Any APM can open the Streamlit sidebar, search by client name, facility name, or date range, and download the original PDF within seconds.

---

## 3. Goals and Objectives

### Primary Goal
Enable any APM to locate and download any past report through a searchable interface in under 60 seconds.

### Secondary Goals
- Eliminate duplicate report generation caused by inability to find previous audits
- Durable server-side record of all audit activity including generating APM's name
- Foundation for future reporting, analytics, and access controls

### Non-Goals (Explicit)
- User authentication or role-based access control
- Report versioning or edit history
- In-browser PDF preview (download only)
- Automated retention/expiration policies (reports kept indefinitely)
- Audit trail logging (who viewed or downloaded)
- Email or notification functionality
- Bulk download or bulk delete
- Cloud storage integration (local VPS filesystem only)

---

## 4. Target Users

### Primary: Asset Protection Manager (APM)
Generates reports back-to-back across multiple facilities. Needs to retrieve past reports to respond to client questions or prepare for re-audits.

### Secondary: APM Team Lead / Reviewer
Reviews completed reports for quality and compliance. May not have generated the report themselves.

---

## 5. Proposed Solution

At PDF generation completion, the system automatically writes a metadata record to SQLite (`data/reports.db`) and saves the PDF to `reports/`. A "Past Reports" sidebar section in Streamlit exposes the library with search, date filtering, download, and delete capabilities.

### Key Capabilities
1. **Automatic cataloging** — zero extra steps from APM
2. **Sidebar search** — live-filtering by client/facility name + date range
3. **Download on demand** — streams PDF directly to browser
4. **Manual deletion** — with confirmation step
5. **Recent reports view** — defaults to 10 most recent

## 7. Technical Specifications

### 7.1 Technology Constraints

| Constraint | Detail |
|---|---|
| Language | Python 3.x |
| UI Framework | Streamlit |
| Database | SQLite via Python `sqlite3` stdlib |
| PDF Storage | Local VPS filesystem (`reports/`) |
| Deployment | Hostinger VPS, shared access, single instance |

### 7.2 Database Schema

```sql
CREATE TABLE IF NOT EXISTS reports (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    facility_name   TEXT    NOT NULL,
    client_name     TEXT    NOT NULL,
    apm_name        TEXT    NOT NULL,
    date_generated  TEXT    NOT NULL,  -- ISO 8601 UTC: YYYY-MM-DDTHH:MM:SS
    file_path       TEXT    NOT NULL UNIQUE
);

CREATE INDEX IF NOT EXISTS idx_reports_client_name
    ON reports (client_name COLLATE NOCASE);

CREATE INDEX IF NOT EXISTS idx_reports_facility_name
    ON reports (facility_name COLLATE NOCASE);

CREATE INDEX IF NOT EXISTS idx_reports_date_generated
    ON reports (date_generated);
```

### 7.3 Module Interface (`src/document_library.py`)

```python
def init_db() -> None:
    """Create data/ directory, reports.db, and reports table if not present."""

def save_report(
    facility_name: str,
    client_name: str,
    apm_name: str,
    file_path: str,
) -> int:
    """Persist report metadata. Returns new record ID."""

def search_reports(
    client_name: str | None = None,
    facility_name: str | None = None,
    date_from: str | None = None,   # YYYY-MM-DD
    date_to: str | None = None,     # YYYY-MM-DD
) -> list[dict]:
    """Return matching records as list of dicts."""

def get_report_by_id(report_id: int) -> dict | None:
    """Return single record dict or None if not found."""

def delete_report(report_id: int) -> bool:
    """Delete DB record and associated PDF. Returns True on success."""
```

### 7.4 File Naming Convention

```python
import re
from datetime import date

def _slugify(value: str) -> str:
    value = value.lower().strip()
    value = re.sub(r'[^\w\s-]', '', value)
    value = re.sub(r'[\s_-]+', '_', value)
    return value

def build_filename(facility_name: str, client_name: str) -> str:
    return (
        f"{date.today().isoformat()}"
        f"_{_slugify(facility_name)}"
        f"_{_slugify(client_name)}"
        ".pdf"
    )
```

### 7.5 Call Sequence — Report Save Flow

```
APM completes report form
        ↓
PRD-01: collect apm_name (text input)
        ↓
PRD-02: build_pdf() → returns file_path (str)
        ↓  [only on success]
document_library.save_report(facility_name, client_name, apm_name, file_path)
        ↓
Sidebar refreshes — new report at top of Past Reports
```

`save_report` must only be called after `build_pdf` returns successfully. Failed PDF assembly must not write a DB record.

### 7.6 Concurrency

WAL mode applied at connection time:
```python
conn.execute("PRAGMA journal_mode=WAL;")
```

Each function opens and closes its own connection (or uses context manager) — no persistent module-level connection.

### 7.7 Error Handling

| Scenario | Behavior |
|---|---|
| `save_report` called with duplicate `file_path` | Raise `ValueError` with descriptive message (UNIQUE constraint safety net) |
| `delete_report` — file not on disk | Log WARNING, delete DB record, return `True` |
| `delete_report` — DB record not found | Return `False`, do not raise |
| `search_reports` — DB not initialized | Call `init_db()` implicitly, return empty list |
| File read error during download | Surface `st.error()`: "Could not read report file. Contact your administrator." |

## 8. User Stories

**US-001 — Automatic report cataloging on generation**
Given PDF assembly succeeds, a DB record is inserted with all metadata. Given assembly fails, no record is inserted. New report appears at top of sidebar without page reload.

**US-002 — View recent reports in sidebar**
Default: 10 most recent reports ordered by date descending. Each row shows client name, facility name, date (YYYY-MM-DD), APM name. "No reports found" message when library is empty.

**US-003 — Search reports by client or facility name**
Real-time filtering on keystroke. Partial, case-insensitive match. Clearing search returns to 10 most recent. "No reports found" when no matches.

**US-004 — Filter reports by date range**
From/To dates filter inclusively. Either date alone sets an open-ended bound. Combined with name search uses AND logic.

**US-005 — Download a past report**
Download button streams PDF to browser. Filename matches stored filename. Broken file_path shows "File unavailable" disabled label.

**US-006 — Delete a report from the library**
"Delete" → inline confirmation → "Confirm Delete" removes DB record and PDF → sidebar refreshes. "Cancel" dismisses with no changes. Missing file on deletion: DB record still removed, no error shown.

**US-007 — APM name captured at report generation**
"APM Name" text input present and required on the form (PRD-01 integration). Blank name blocks submission. Stored value is trimmed of whitespace.

**US-008 — Automatic DB and directory initialization**
Fresh VPS: both `data/` and `reports/` directories and `reports.db` created on first app start with no manual intervention. Restart on existing DB does not drop or alter data.

**US-009 — Gitignore protection**
Running `git status` after generating a report shows no files under `reports/` or `data/`.

## 11. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| VPS disk fills up over time | Medium | High | Delete functionality enables manual cleanup; document process; monitor disk usage |
| Concurrent SQLite write conflict | Low | Medium | WAL mode handles concurrent reads; short write duration minimizes risk |
| APM enters another APM's name | High | Low | Accepted risk; self-reported and trust-based per Non-Goals |
| `file_path` invalid after VPS migration | Medium | Medium | **Open question: relative vs absolute path storage** |
| Filename collision (same facility/client/date) | Low | Medium | UNIQUE constraint prevents silent overwrite; **Open question: collision resolution (timestamp suffix or UUID?)** |
